## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser order payload -> server action | Submitted order can include forged IDs, duplicates or cross-duo games. |
| Library status -> play role creation | Moving to `jogando` must not bypass existing status and limit rules. |
| Client drag state -> authoritative order | Client order is presentation until server validates and commits. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Tampering | Reorder payload | mitigate | Validate membership, active status, exact active set and unique IDs before writing. |
| T-04-02-02 | Information Disclosure | Dashboard read model | mitigate | Read through RLS-scoped user transaction and never accept `duoId` from client. |
| T-04-02-03 | Denial of Service | Dashboard read | mitigate | Batch catalog hydration and avoid unbounded library scans. |
| T-04-02-04 | Elevation of Privilege | Promotion | mitigate | Server action requires authoritative verified session and DB enforces one Principal. |
</threat_model>
