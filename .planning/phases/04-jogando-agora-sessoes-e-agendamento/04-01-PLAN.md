<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> play mutations | Client-supplied IDs, durations, confirmations, percentages and push endpoints are untrusted. |
| Play module -> Postgres | Runtime role must use RLS and least privilege; migrations use owner/migrator only. |
| Cron/worker -> jobs | Worker invocations may duplicate or overlap and must be idempotent. |
| Push endpoint material -> server adapters | Subscription secrets stay server-side and are redacted in logs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-01 | Spoofing | Session/terminal confirmations | mitigate | Confirming user must be current authenticated user and a duo member; unique confirmation rows block impersonation and duplicates. |
| T-04-01-02 | Tampering | Active game role/order | mitigate | Lock duo/active rows, enforce one principal and position uniqueness in DB, and reject fourth active game server-side. |
| T-04-01-03 | Repudiation | XP/session/status effects | mitigate | Store actor IDs, confirmation timestamps, XP award keys and domain events. |
| T-04-01-04 | Information Disclosure | Spoiler Momentos and push endpoints | mitigate | RLS, local reveal rows, server-only push material and endpoint redaction in logs. |
| T-04-01-05 | Denial of Service | Reminder jobs | mitigate | Due-job indexes, batch limits, locks and idempotency; no unbounded job scans. |
| T-04-01-06 | Elevation of Privilege | Cross-duo mutations | mitigate | Forced RLS plus application authorization and integration tests for every new table family. |
</threat_model>
