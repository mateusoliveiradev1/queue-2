# Phase 03.1: Refinos Visuais e UX da Descoberta - Research

**Researched:** 2026-06-04
**Status:** Complete
**Question:** What do I need to know to plan this phase well?

## Executive Summary

Phase 03.1 should be planned as a focused product-quality intervention over the existing Phase 3 Discovery implementation, not as new Discovery domain scope. The implementation already has the required server actions, domain rules, persistence, search, live session, mood quiz, match history, handoff and E2E scaffolding. The phase should therefore split work into four concerns:

1. Fix the known architecture inversion so `discovery/application` no longer imports `discovery/presentation/view-models`, and extend `check:architecture` to reject same-module layer regressions.
2. Reshape `/app/descobrir` so the first fold is a deck-dominant, Tinder-like QUEUE/2 card ritual instead of a dashboard of filters, modes and side panels.
3. Polish mobile shell, control feedback, motion, reduced-motion equivalents, search overlay, match celebration and handoff while preserving server-authoritative decisions.
4. Turn authenticated browser validation and fixture documentation into blocking evidence before Phase 4.

The planner should keep this phase narrow: do not add Play sessions, roulette, audio, Hall/reviews, new database schema, or new discovery semantics unless a test exposes a regression in existing behavior.

## Current Implementation Findings

### Route composition

`apps/web/src/app/app/descobrir/page.tsx` already uses the expected authenticated route pattern:

- `requireVerifiedSession()` protects the route.
- pairing/naming states redirect before data rendering.
- reads go through discovery public APIs from `apps/web/src/modules/discovery/index.ts`.
- server actions are passed as props to presentation components.
- `returnTo` and route params preserve filter/live/surprise state.

The current layout conflicts with the Phase 03.1 context because the first fold renders:

- a full route header with explanatory copy,
- a row of four mode buttons,
- a full filter form,
- a `surface-band` deck shell,
- a right side rail with Live, Search, Quiz and History panels.

This makes the card a section among sections, not the product surface.

### Discovery presentation components

Useful components already exist and should be refactored rather than replaced wholesale:

- `DiscoveryDeck` already supports drag, keyboard decisions and `useReducedMotion`.
- `DiscoveryCard` centralizes cover, source metadata, recommendation reasons, decision forms and handoff actions.
- `MatchCelebration` exists but is currently a restrained status panel; Phase 03.1 must make it a real match event.
- `DiscoverySearch`, `DiscoveryFilters`, `LivePanel`, `MoodQuiz` and `MatchHistory` contain the required behaviors but need to orbit the card as compact controls, trays or sheets.

The high-risk UI issue is not missing data; it is hierarchy. The route currently gives equal visual weight to modes, filters, cards, panels and history.

### Architecture debt

The exact debt from context is present:

- `apps/web/src/modules/discovery/application/get-discovery-deck.ts`
- `apps/web/src/modules/discovery/application/search-discovery-games.ts`
- `apps/web/src/modules/discovery/application/get-surprise-recommendation.ts`

all import builders from `../presentation/view-models`.

`scripts/check-architecture.mjs` currently catches:

- cross-module deep imports,
- domain impurity,
- Client Component server-only imports,
- shared package imports from app code,
- workspace dependency mistakes.

It does not yet catch same-module dependency direction such as `application -> presentation`. Phase 03.1 should add same-module layer-order checks. The allowed local direction should be:

`presentation -> application -> domain`

and infrastructure may depend on application ports and domain, but application must not import presentation.

The lowest-risk fix is to move discovery card/fact builders into an application-safe read-model file such as `apps/web/src/modules/discovery/application/view-models.ts` or `apps/web/src/modules/discovery/application/deck-view-models.ts`, then have presentation import types/data from application rather than the reverse. Avoid moving business rules into React components.

### CSS and app shell

The brand tokens are already available through `packages/ui/src/tokens.css`:

- `--primary` acid lime,
- `--accent` violet,
- `--font-display`, `--font-body`, `--font-mono`,
- `--radius-sharp`,
- `--focus-ring`.

`apps/web/src/app/globals.css` has extensive Discovery CSS starting around the Discovery section. It already uses sharp radii, OKLCH color mixing, focus outlines and some reduced-motion behavior. Mobile behavior is still mostly "collapse desktop into one column":

- `.app-shell` switches from sidebar grid to one column under 820px.
- `.app-sidebar` becomes a top block, not a thumb-friendly bottom navigation.
- `.discovery-route-grid`, `.discovery-mode-actions`, `.discovery-card` collapse to one column.

Phase 03.1 should add a real mobile authenticated nav pattern, with bottom navigation and compact active states, while preserving the desktop rail. This is shared foundation work, but its scope should be limited to what Discovery needs.

### Test surface

Existing tests are useful but encode the old design:

- `apps/web/tests/discovery-ui.test.tsx` expects the row of `Live`, `Surpresa`, `Quiz` and `Busca` before the deck and tests the full filter form.
- `apps/web/tests/phase-3-e2e.spec.ts` checks deck/modes/filters/search/live/handoff, but with old visible locations and old route heading.
- `apps/web/tests/accessibility.spec.ts` already checks `/app/descobrir`, reduced motion, keyboard decisions, source links and push opt-in when fixtures exist.
- `apps/web/tests/discovery-application.test.ts` can be extended to assert the architecture refactor indirectly, and `check:architecture` should provide the hard gate.

The planner should require tests to be rewritten around the new product hierarchy, not just relaxed.

## External Design Rubric

The phase context references Impeccable as a rubric, not as a package to install. The public repository describes it as a frontend design vocabulary with domain references for typography, color/contrast, spatial design, motion, interaction, responsive design and UX writing. It also explicitly calls out anti-patterns relevant to this phase: generic SaaS templates, over-carded layouts, nested cards, gray-on-colored text, overused gradients and weak default visual choices.

Source: https://github.com/pbakaus/impeccable

Planning implication: use Impeccable as acceptance language:

- `shape`: deck-first hierarchy and card-stage composition before editing details.
- `animate`: purposeful swipe, reveal, match and transition motion with reduced-motion parity.
- `clarify`: short PT-BR ritual copy and clear state labels.
- `adapt`: mobile bottom nav, thumb targets and card-first mobile view.
- `audit`: browser checks for overlap, focus, hydration, reduced motion and broken controls.
- `polish`: final pass over text fit, button states, source/freshness and anti-patterns.

Do not install or vendor Impeccable in this phase.

## Planning Strategy

### Recommended plan split

Use four execution plans:

1. **Architecture and read-model boundary**
   - Move discovery deck/card builders out of `presentation`.
   - Keep domain semantics and repository behavior unchanged.
   - Extend `scripts/check-architecture.mjs` to block same-module layer inversions.
   - Update application tests and architecture fixtures.

2. **Deck-first Discovery stage and app shell**
   - Restructure `/app/descobrir` around a dominant central card stage.
   - Convert mode buttons and filters into compact orbit controls.
   - Introduce mobile bottom navigation in `AppShell`.
   - Refine desktop rail active state and footer.

3. **Interaction, motion, search, match celebration and handoff polish**
   - Upgrade `DiscoveryDeck`, `DiscoveryCard`, `DiscoverySearch`, `DiscoveryFilters`, `LivePanel`, `MoodQuiz`, `MatchCelebration`, `MatchHistory`.
   - Add pending/success/error/disabled/focus/pressed states.
   - Make search an overlay/sheet whose selected result becomes the focused card context.
   - Keep handoff after match or in secondary card tray; do not let Wishlist/Jogando/Pausado compete with the three deck decisions.

4. **Browser validation and setup evidence**
   - Update Vitest, Playwright and accessibility expectations for the new hierarchy.
   - Run feasible local checks.
   - Add a Discovery setup/runbook section or doc covering migrations, VAPID, `TEST_DATABASE_URL`, `E2E_BASE_URL`, ready duo fixtures and `E2E_PHASE3_DISCOVERY_QUERY`.
   - Record explicit fixture skips as gates, not passes.

This split keeps architecture unblocked before UI polish, then makes browser validation the final gate.

## UX Requirements to Preserve

Decision semantics must remain unchanged:

- `Quero jogar` records `want`, positive preference, approval, card removal, and match creation only when partner already wanted the same game.
- `Agora nao` records `not_now`, negative preference, 14-day cooldown, card removal, no match approval.
- `Pular` records `skip`, zero preference, no cooldown, no approval/rejection, card advance.

The UI may add optimistic and pending feedback, but final state must remain server-authoritative and failures must leave clear status.

Library handoff must remain explicit and limited to Wishlist, Jogando and Pausado. `Zerado` and `Dropado` remain disabled until Phase 4.

RAWG attribution/source/freshness must remain visible when RAWG data/images appear.

## Validation Architecture

### Dimensions

1. **Architecture boundary validation**
   - `pnpm check:architecture` fails when any module imports a higher-level same-module layer, especially `application -> presentation`.
   - Application tests continue to cover discovery use-case behavior after read-model relocation.

2. **Decision behavior validation**
   - `pnpm --filter @queue/web test -- discovery-domain discovery-application discovery-search discovery-ui` covers `want`, `not_now`, `skip`, search, surprise, and handoff.
   - Existing DB integration tests remain the evidence for RLS and concurrency when `TEST_DATABASE_URL` is configured.

3. **UI structure and interaction validation**
   - Component tests assert deck-first hierarchy, compact mode controls, secondary filters/search, card decisions, match celebration and blocked future statuses.
   - Tests should fail if the old row of four giant mode buttons or first-fold filter dashboard returns.

4. **Accessibility and reduced-motion validation**
   - `apps/web/tests/accessibility.spec.ts` covers keyboard decisions, source links, focus visibility and reduced-motion mode.
   - Reduced-motion assertions should verify equivalent status/copy, not just disabled animation.

5. **Authenticated browser validation**
   - Playwright must run against a real authenticated duo fixture when environment exists.
   - Required viewports: desktop, mobile, and reduced motion.
   - Required checks: no hydration errors, no text overlap, visible focus, usable controls, match celebration, search overlay, deck decisions, mobile bottom nav.

6. **Setup documentation validation**
   - `.env.example` and/or a runbook must make migrations, VAPID keys, `TEST_DATABASE_URL`, `E2E_BASE_URL`, ready duo accounts and `E2E_PHASE3_DISCOVERY_QUERY` explicit.
   - Missing fixture output remains a blocker to record, not passing evidence.

### Commands

- `pnpm check:architecture`
- `pnpm --filter @queue/web typecheck`
- `pnpm --filter @queue/web test -- discovery-domain discovery-application discovery-search discovery-ui`
- `pnpm --filter @queue/web test -- accessibility` where applicable
- `pnpm --filter @queue/db test:integration -- discovery` when `TEST_DATABASE_URL` is configured
- `pnpm --filter @queue/web test:e2e -- tests/phase-3-e2e.spec.ts` when `E2E_BASE_URL` and ready fixtures are configured
- Browser manual/agent review against `/app/descobrir` for desktop, mobile and reduced motion

## Risks And Mitigations

| Risk | Mitigation |
|------|------------|
| Visual polish hides decision semantics | Keep domain tests unchanged and require server-authoritative result handling in UI tasks. |
| Search overlay duplicates Catalog | Acceptance criteria must require selected result to become focused Discovery card context. |
| Mobile shell refactor destabilizes other routes | Limit shared shell changes to navigation/responsive foundation and keep broad app route smoke tests. |
| Motion blocks accessibility | Require `useReducedMotion`, keyboard path and reduced-motion browser review in every motion task. |
| Architecture checker creates false positives | Implement layer-order checks with explicit module layer metadata and add targeted fixture/source tests. |
| Browser validation silently skips | Require setup documentation and final summary to classify missing fixtures as gates, not passes. |

## Recommended PLAN Acceptance Themes

- The first viewport of `/app/descobrir` is card-dominant on desktop and mobile.
- `Live`, `Surpresa`, `Quiz`, `Busca`, filters, history and handoff orbit the deck.
- All Discovery controls expose hover, focus-visible, pressed, pending/loading, success/error, disabled and reduced-motion states where applicable.
- Match celebration is a real event with game cover, `/2`, pointer motif and short PT-BR copy such as `Os dois quiseram`.
- No generic confetti, random particles, casino/arcade spectacle, nested-card layout or giant filter dashboard.
- Browser evidence is mandatory before Phase 4.
