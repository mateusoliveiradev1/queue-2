## Phase 03.1 Screen Contract

`/app/descobrir` must stop reading as an admin dashboard. The first viewport is a Discovery ritual:

| Surface | Contract |
|---------|----------|
| Primary focal point | One dominant active game card, with cover art and decision gesture visible before filters or panels. |
| First fold | Deck stage first; route heading, filters, mode controls, history, search, quiz, and live state are compact orbit controls. |
| Card layers | First layer is visual and decisive: cover, title, compatibility signal, and `Quero jogar`, `Agora nao`, `Pular`. Secondary tray carries platform, tempo estimado, availability, source/freshness, and library handoff. |
| Desktop shell | Sticky rail can remain, but active state is clearer, grouping is quieter, and footer/status copy does not compete with Discovery. |
| Mobile shell | Use bottom navigation with clear active state and thumb-friendly targets. Discovery card occupies almost the whole experience. |
| Search | Opens as an overlay or sheet over the card stage. Selecting a result turns it into the focused Discovery card context. |
| Filters | Quick filters are compact chips/controls. Advanced filters belong in a sheet/drawer or secondary layer. |
| Supporting modes | `Live`, `Surpresa`, `Quiz`, `Busca`, history, and handoff orbit the deck; no row of four giant mode buttons before the card. |
| Match celebration | A match is an editorial event with cover art, `/2`, pointer motif, short PT-BR copy, and meaningful lime/violet accents. It must not be only a toast. |

**Blocked visual patterns:** giant first-fold filter forms, rows of equal CTAs competing with the card, nested/floating card stacks, generic neon/terminal styling, casino/arcade spectacle, meaningless glow, random particles, clipped labels, invisible focus, and mobile layouts where the card is pushed away from the first action.

## Spacing Scale

Declared values for new Phase 03.1 layout decisions (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, hairline offsets, card badge inner gaps |
| sm | 8px | Compact control gaps, chip spacing, grouped metadata |
| md | 16px | Default element spacing, card inner sections, form rows |
| lg | 24px | Stage padding, sheet padding, mode/control clusters |
| xl | 32px | Desktop deck and orbit-control gaps |
| 2xl | 48px | Major section breaks and desktop card-stage breathing room |
| 3xl | 64px | Page-level max separation only where the card remains first-fold dominant |

Exceptions: none for spacing. Fixed control dimensions may use minimum touch targets of 44px or higher, but those are dimensions, not spacing tokens. Existing global tokens outside this set should not be introduced into new Discovery layout work unless refactored into the standard set.

## Typography

Use exactly these four size roles and exactly these two weights for Phase 03.1 UI work.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 600 | 1.35 |
| Body | 16px | 400 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.1 |

Typography rules:

- Do not scale font size with viewport width.
- Letter spacing is `0`, including labels and display type.
- Display text is reserved for the active game title, match event headline, and major card-stage moments.
- Compact controls, sheets, side rails, and bottom nav use Label or Body sizes only.
- If text does not fit, change layout, line breaks, or max width; do not shrink below Label.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--bg)` / `oklch(0.16 0.025 285)` | App background, deep stage, full-viewport negative space |
| Secondary (30%) | `var(--surface)` / `oklch(0.21 0.03 285)` and `var(--surface-raised)` / `oklch(0.26 0.035 285)` | Card bodies, sheets, bottom nav, desktop rail, secondary trays |
| Accent (10%) | `var(--primary)` / `oklch(0.86 0.22 128)` | Positive decision, primary CTA, active nav marker, focus ring, `/2`, pointer motif |
| Event accent | `var(--accent)` / `oklch(0.62 0.27 305)` | Match celebration edge, partner/live state, surprise reveal, quiz result emphasis |
| Destructive | `var(--danger)` / `oklch(0.68 0.22 25)` | Errors and truly destructive confirmations only |

Accent reserved for: `Quero jogar`, primary positive pending/success feedback, active route marker, keyboard focus ring, `/2`, pointer motif, match event highlight, surprise reveal, and quiz result emphasis. Accent is never used for every interactive element.

Color rules:

- `Agora nao` is a temporary/no-guilt cooldown state. Use muted warning-adjacent treatment, not destructive red.
- `Pular` is neutral. Use surface/ink contrast without primary accent.
- Violet is semantically reserved for event/partner/surprise moments; it is not a second default accent.
- Rarity colors remain meaningful metadata and future roulette/achievement language, not general decoration.
- RAWG/source/freshness metadata must remain readable and visually subordinate to decisions.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Quero jogar` |
| Secondary decision | `Agora nao` |
| Neutral decision | `Pular` |
| Empty state heading | `A fila de descoberta respirou.` |
| Empty state body | `Ajustem os filtros ou voltem mais tarde para puxar outro coop para a dupla.` |
| Search empty state | `Nada entrou na fila.` Body: `Tente outro nome ou limpe os filtros antes de buscar de novo.` |
| Error state | `Nao deu para mover a carta.` Body: `Sua decisao nao foi salva. Tente de novo antes de seguir.` |
| Match headline | `Os dois quiseram` |
| Match body | `Entrou no radar da dupla. Escolham para onde esse jogo vai agora.` |
| Handoff CTA - wishlist | `Mandar para Wishlist` |
| Handoff CTA - playing | `Comecar em Jogando` |
| Handoff CTA - paused | `Guardar em Pausado` |
| Destructive confirmation | none in this phase; Discovery decisions are reversible or non-destructive except server-authoritative cooldown for `Agora nao` |

Copy rules:

- Primary decision labels stay `Quero jogar`, `Agora nao`, and `Pular`.
- PT-BR copy is short, ritualistic, and owned by QUEUE/2.
- Avoid generic SaaS copy such as `acao concluida com sucesso`, `submit`, `ok`, `cancel`, and `no data found`.
- Error copy must say what failed and how the user can recover.
- Handoff after a match may offer Wishlist, Jogando, or Pausado only. `Zerado` and `Dropado` stay blocked until Phase 4.

## Interaction And Motion Contract

| Moment | Contract |
|--------|----------|
| Drag/swipe | Card follows pointer with slight tilt and visible decision state. Keyboard path triggers the same decision semantics. |
| Decision submit | Show pressed and pending feedback immediately. Final state is server-authoritative; failures keep the user oriented. |
| Card exit | `Quero jogar`, `Agora nao`, and `Pular` each have distinct but restrained exit treatment. |
| Next card | Next card enters with continuity so the ritual feels uninterrupted. |
| Match | Match celebration interrupts the deck as a real event, then reveals handoff actions. |
| Surprise | Reveal feels intentional and lower-spectacle than future roulette. |
| Quiz | Selection/result feedback is visible, compact, and connected to the card stage. |
| Search | Overlay supports typing, loading, empty, error, selected, focus, keyboard navigation, and reduced motion. |
| Loading | Use QUEUE/2 `/2` stroke loading or a brand-consistent skeleton, not a generic spinner. |

Reduced-motion equivalent:

- Replace heavy drag/exit/match choreography with instant state changes, short fades, visible status copy, and the same decision/handoff information.
- Reduced motion is not a degraded route. It must still communicate decide, reveal, confirm, celebrate, and orient.
- No sound in this phase.

## Control States And Accessibility

Every Discovery control must declare and implement:

| State | Contract |
|-------|----------|
| Hover | Visible surface or border change without relying only on glow. |
| Focus-visible | `var(--focus-ring)` or stronger equivalent with visible outline at desktop and mobile sizes. |
| Pressed | Tactile state for buttons, chips, card actions, and bottom-nav items. |
| Pending/loading | Disabled duplicate submit path plus clear in-control or nearby status. |
| Success | Short positive feedback tied to the action, not a generic toast-only response. |
| Error | Action-specific problem and retry path. |
| Disabled | Readable label, visible reason when status is blocked, and no hidden pointer target. |
| Reduced motion | Same meaning with lower-motion state change. |

Icon-only actions are allowed only when they have an accessible name, a visible label on small screens or focus/hover, and a stable 44px minimum target.

- Do not put business rules in routes, React components, or CSS.
- Do not move RAWG attribution/source/freshness out of sight.

- Update tests so the old dashboard layout fails if it returns.
