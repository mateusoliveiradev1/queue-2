# Phase 02.1: Localizacao e Qualidade do Catalogo - Context

**Gathered:** 2026-06-03T20:25:15.9689291-03:00
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase turns the Phase 2 RAWG-backed catalog into a Portuguese-Brazilian catalog that is reliable enough to support discovery. It defines and implements a reviewed PT-BR localization strategy, persistent localization cache, repeatable RAWG synchronization path, and source/freshness UI that explains what came from RAWG, what came from QUEUE/2 curation, and what is unavailable.

The phase must not weaken the Phase 2 catalog shape: the main flow still prioritizes confirmed two-player campaign/story coop games, RAWG remains the source for external data and images with active attribution, and secrets stay server-only. The phase does not implement Phase 3 swipe/match behavior, a broad public catalog import, user-submitted catalog edits, or a mandatory production cron.

</domain>

<decisions>
## Implementation Decisions

### Fonte da localizacao PT-BR

- **D-01:** The localization strategy is hybrid with review: important games can have curated QUEUE/2 seed text, and provider-assisted translation may exist only as draft input. Nothing appears in the product until reviewed and cached.
- **D-02:** Do not lock a translation provider in this discussion. The researcher/planner must compare options such as DeepL, OpenAI, Google or other suitable providers against cost, PT-BR quality, operational fit, terms, secret handling, and review workflow before choosing.
- **D-03:** Descriptions are never translated at page runtime. Runtime pages read published cached localization or show an honest PT-BR fallback state.
- **D-04:** If a game has no reviewed PT-BR description, the UI must not show raw English RAWG description. It should say that the Portuguese description is not reviewed yet or use equivalent honest PT-BR copy.
- **D-05:** Publishing a PT-BR description requires a quality checklist: correct two-player coop facts, no heavy spoilers, no invented facts, natural Brazilian Portuguese, contained QUEUE/2 tone, and recorded source/author/reviewer metadata.
- **D-06:** QUEUE/2 curation controls the displayed description text. RAWG remains the external source for data and images, but reviewed QUEUE/2 copy is the product-facing description.
- **D-07:** Only internal operation/admin technical paths may approve or publish PT-BR descriptions in this phase. Duo members do not edit or suggest catalog descriptions.
- **D-08:** The writing tone is informative with restrained QUEUE/2 personality. It should explain coop gameplay clearly without becoming a review, hype copy, jokes, or a generic encyclopedia entry.
- **D-09:** This phase prioritizes descriptions. Genre/platform labels may get simple PT-BR mappings when they would otherwise appear raw or awkward, but full localization of every catalog label is not the primary scope.

### Cache e modelo de dados

- **D-10:** Reviewed descriptions live in a dedicated localization table, e.g. `catalog.game_localizations`, rather than in `catalog.games`. The table should include locale, text, status, source/provenance, author/reviewer metadata, timestamps, and versioning.
- **D-11:** Localization status must support at least `draft`, `review`, `published`, and `rejected`.
- **D-12:** Localization history is versioned. The product reads the newest valid `published` localization for the requested locale, while prior versions remain available for audit and rollback.
- **D-13:** `catalog.games.description` remains separate as RAWG source data. It is not copied into localization rows by default and is not displayed as a fallback when PT-BR localization is missing.
- **D-14:** The web runtime reads published localizations only. Writes to localization drafts, review transitions, and publication belong to worker/admin technical credentials, not normal browser-triggered runtime mutations.

### Sincronizacao RAWG

- **D-15:** Phase 02.1 should provide a versioned command/worker path for catalog sync with allowlist/seeds, dry-run support, structured logs, and transactional upserts.
- **D-16:** The initial sync input is a curated allowlist of confirmed or candidate two-player coop games for the QUEUE/2 ritual. RAWG enriches those games; broad RAWG search results do not automatically enter the main catalog.
- **D-17:** RAWG sync may update RAWG-owned facts such as name, cover, platforms, genres, source timestamps, and source URL. It must not overwrite QUEUE/2 curation such as coop confirmation, main-flow eligibility, PT-BR localization, curated time estimates, or curated availability.
- **D-18:** Sync runs and failures should be auditable through `ops`-owned persistence or equivalent structured operational records: run id, timestamps, item counts, per-item errors, and relevant changes.
- **D-19:** This phase prepares the sync path for a future cron, but a daily cron is not mandatory for Phase 02.1 completion.
- **D-20:** `RAWG_API_KEY` and any future translation-provider secret stay server-only. Client components and browser bundles must not import secret-bearing adapters.

### Rotulos de fonte/frescor na UI

- **D-21:** Game detail should use a compact "Fontes e frescor" block with separate lines/categories for RAWG data/images, PT-BR description, estimated time, and availability.
- **D-22:** The published PT-BR description source label is `Descricao revisada: QUEUE/2`.
- **D-23:** When no PT-BR description is published, both the body and the source/freshness block show an honest PT-BR unavailable state. The body must not show raw English.
- **D-24:** Missing estimated time and availability keep the current explicit language: `Sem fonte confiavel ainda` and `Nao verificado`, with copy indicating no active source exists.
- **D-25:** Catalog cards show minimal source/freshness metadata. Game detail shows the complete source/freshness breakdown.
- **D-26:** Freshness dates should be scannable and accessible: relative text such as `Atualizado ha 3 dias` plus an accessible absolute date, for example through semantic `datetime` or equivalent visible/a11y text.
- **D-27:** Fresh, stale, and missing states may use subtle visual tones such as `data-freshness`, but meaning must always be present in text and never depend only on color.
- **D-28:** Author/reviewer names are not shown to end users. They remain internally auditable in the localization model.
- **D-29:** RAWG attribution always has an active link. PT-BR description links appear only when there is a relevant external source; QUEUE/2-owned curation does not need an artificial link.

### the agent's Discretion

- Choose exact table and column names, indexes, constraints, CLI/job names, source enum names, and copy variants as long as they preserve the decisions above.
- Choose the translation provider after research if the implementation includes provider-assisted draft generation; otherwise keep provider integration behind a port so the first implementation can ship with curated seeds and the workflow still allows future provider drafts.
- Decide how much admin/operator UI, if any, is necessary for this phase. A script, seed, or worker path is sufficient if it is repeatable, auditable, and documented.
- Use recommended defaults for any minor unanswered implementation detail: favor least privilege, explicit provenance, no runtime translation, no English fallback, and quiet UI transparency.

</decisions>

## Existing Code Insights

### Reusable Assets

- `apps/web/src/modules/catalog/domain/catalog-policy.ts` already centralizes RAWG attribution, freshness, time estimate and availability state decisions. Extend this rather than moving source rules into routes.
- `apps/web/src/modules/catalog/presentation/source-metadata.tsx` provides a compact source/freshness rendering pattern that can evolve into the detail-level "Fontes e frescor" block.
- `apps/web/src/modules/catalog/presentation/view-models.ts` already decides description and source labels for cards/detail. It is the current presentation seam for replacing hardcoded descriptions with persisted localization.
- `apps/web/src/modules/catalog/infrastructure/rawg-client.ts` is marked `server-only` and already normalizes RAWG responses into catalog upsert input.
- `apps/web/src/modules/catalog/infrastructure/catalog-repository.ts` already uses transactional upserts for catalog facts and can be extended carefully for localization reads and sync writes.
- `packages/db/src/schema/catalog.ts` and migration `0004_catalog_source.sql` already establish `catalog` as external/sourced catalog ownership with worker write privileges.

### Established Patterns

- Routes compose module public APIs and should not hold catalog/localization business rules.
- Domain rules are tested without Next.js, React, Drizzle or external SDKs.
- External source data stays visible and attributed, but product-facing decisions such as main-flow eligibility and curation are QUEUE/2-owned.
- Runtime web access is least-privileged. Worker/admin paths can mutate sourced catalog data; browser-triggered runtime paths should not publish localization.
- Phase 2 UI uses quiet metadata rather than loud badges. Phase 02.1 should preserve that density.

### Integration Points

- Add localization read behavior behind the `catalog` module public API so `/app/catalogo` and `/app/jogo/[slug]` do not deep-import infrastructure.
- Add database schema/migration for localization under the `catalog` schema with explicit comments, constraints, indexes and privileges.
- Add a sync command/worker path outside Client Components, using server-only RAWG and optional translation provider adapters.
- Add operational audit persistence under `ops` or a clearly owned operational table if the existing `audit_events` shape is too duo-scoped for catalog sync.
- Update tests in `apps/web/tests/catalog-domain.test.ts`, `apps/web/tests/catalog-library-ui.test.tsx`, and database integration tests for schema, privileges and no client secret exposure.

</code_context>

<specifics>
## Specific Ideas

- Use the user-facing label `Descricao revisada: QUEUE/2` for published PT-BR catalog descriptions.
- Add a compact detail page block named `Fontes e frescor`.
- Keep catalog cards minimal: RAWG/freshness remains discreet, while the detail page carries the full provenance breakdown.
- Use the fallback idea `Descricao em portugues ainda nao revisada` or equivalent when no published PT-BR text exists.
- Treat current hardcoded descriptions as a temporary seed source to migrate into persisted localization.

</specifics>
