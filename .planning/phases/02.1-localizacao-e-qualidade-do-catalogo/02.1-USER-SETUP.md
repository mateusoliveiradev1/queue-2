## Automatic Catalog Refresh

Production refresh is automatic after deploy on Vercel:

- `vercel.json` schedules `/api/jobs/catalog/refresh` once per day.
- Vercel Cron calls the route in the production deployment.
- The route rejects requests unless `Authorization: Bearer $CRON_SECRET` matches.
- The job runs RAWG sync, reapplies QUEUE/2 curation seeds, then validates published PT-BR descriptions and verified availability rows.

Manual execution is only for local debugging or force-running outside the daily schedule:

```bash
pnpm catalog:refresh
```

For local use, keep the Next app running and set `CRON_SECRET` to the same value used by the route. Use `CATALOG_REFRESH_URL` only when the target is not `http://localhost:3000/api/jobs/catalog/refresh`.

## Verification

```bash
pnpm catalog:seed-curation -- --apply
```

Expected result:

- `localizations.appliedCount` matches the synchronized curated catalog seed count.
- `availability.appliedCount` includes only entries with verified source URLs.
- No missing slugs are reported after the curated RAWG allowlist is synchronized.

After setting `RAWG_API_KEY`, run:

```bash
pnpm catalog:sync -- --dry-run
```

Expected result:

- JSON output with `mode: "dry-run"`.
- Planned/failed item counts visible without database mutation.
- No secret values printed.

To verify the full automatic path locally, run the app and then:

```bash
pnpm catalog:refresh
```

Expected result:

- HTTP JSON output from `/api/jobs/catalog/refresh`.
- `sync.failedCount` is `0`.
- `health.ok` is `true`.
- The command exits nonzero if `CRON_SECRET` or required RAWG production sync configuration is missing.

After configuring optional provider keys, run:

```bash
pnpm check:secrets
```

Expected result:

- Secret scan passes.
- No provider key names appear in client source or client bundles.
