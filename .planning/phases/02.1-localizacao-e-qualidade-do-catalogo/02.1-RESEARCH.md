# Phase 02.1: Localizacao e Qualidade do Catalogo - Research

**Researched:** 2026-06-03T23:55:00-03:00
**Status:** Ready for planning
**Question:** What do we need to know to plan Phase 02.1 well?

## Executive Summary

Phase 02.1 should be planned as a reliability and provenance upgrade over the accepted Phase 2 catalog, not as a broad catalog rewrite. The work has four connected parts:

1. Persist reviewed PT-BR localizations in `catalog.game_localizations`, replacing temporary hardcoded presentation descriptions.
2. Make RAWG sync repeatable and auditable through a worker/CLI path that starts from curated coop allowlists and preserves QUEUE/2-owned curation fields.
3. Extend catalog read models and UI source metadata so cards stay compact while game detail shows a full `Fontes e frescor` breakdown.
4. Document the localization provider decision: use curated QUEUE/2 seeds as the first published source, keep provider-assisted translation behind a draft-only port, and prefer DeepL API Growth as the first paid translation candidate if automated draft generation is implemented.

The key product rule is stricter than Phase 2: runtime pages must never translate on demand and must never show raw English RAWG descriptions as a PT-BR fallback. The web runtime reads only published localizations; all draft/review/publish writes belong to worker/admin technical paths.

## Existing Code Baseline

Phase 2 already established the useful shape:

- `packages/db/src/schema/catalog.ts` defines `catalog.games`, platform, genre, time estimate and availability tables, but no localization table.
- `packages/db/src/migrations/0004_catalog_source.sql` grants app runtime read-only access and worker write access for catalog facts.
- `apps/web/src/modules/catalog/infrastructure/rawg-client.ts` is `server-only` and normalizes RAWG payloads into catalog upsert input.
- `apps/web/src/modules/catalog/infrastructure/catalog-repository.ts` reads catalog details and performs transactional upserts.
- `apps/web/src/modules/catalog/presentation/view-models.ts` maps catalog records to card/detail views.
- `apps/web/src/modules/catalog/presentation/localized-descriptions.ts` is a temporary hardcoded PT-BR seed map created by post-UAT polish.
- `apps/web/src/modules/catalog/presentation/source-metadata.tsx` is the current compact source/freshness component.
- `apps/web/src/app/app/jogo/[slug]/page.tsx` is the detail surface that needs the full `Fontes e frescor` block.
- `apps/web/tests/catalog-domain.test.ts` and `apps/web/tests/catalog-library-ui.test.tsx` already cover source labels and PT-BR description preference.
- `packages/db/tests/catalog-schema.test.ts` is the integration-test entry point for catalog schema, constraints and privileges.

Important code risks discovered during research:

- `toCatalogGameDetailView` currently falls back from hardcoded PT-BR text to `game.description`, which can be raw RAWG English. This violates D-03 and D-04 for Phase 02.1 and must be removed.
- `catalogRepository.upsertCatalogGame` currently overwrites `coop_campaign_confirmed`, player count, confirmation source, confirmation checked timestamp and `main_flow_eligible` from the upsert input. Because `normalizeRawgGame` sets those fields to unconfirmed defaults, a sync path built on this as-is can erase QUEUE/2 curation. The plan must separate RAWG-owned updates from QUEUE/2-owned fields.
- `ops.audit_events` and `ops.domain_events` are duo-scoped today, so catalog sync audit should not reuse them directly unless the schema is extended. A catalog-wide `ops.catalog_sync_runs` / `ops.catalog_sync_run_items` pair is cleaner.

## Localization Provider Research

The locked product decision is hybrid-with-review. Provider output is draft input only; publication is a QUEUE/2 decision after checklist review.

| Candidate | Fit for PT-BR catalog drafts | Cost/ops shape | Data/security notes | Planning recommendation |
|-----------|------------------------------|----------------|---------------------|-------------------------|
| Curated QUEUE/2 seeds | Highest control and already exists for known games. | No provider cost; slowest at scale. | No external secret or third-party text transfer. | Use for first published localizations and migrate current `localized-descriptions.ts` into persisted `published` rows. |
| DeepL API | Strong translation-product fit, simple character billing, good quality reputation for localization workflows. | Developer plan has a one-time total character allowance; Growth adds included monthly volume, admin controls and usage limits. | DeepL documents immediate deletion for API Growth/Pro text processing. | Prefer as the first provider-assisted draft adapter if automated drafts are implemented in this phase. |
| Google Cloud Translation Advanced | Mature IAM, glossary, regional/data-location options and custom/adaptive translation. | NMT and custom models are character-metered; LLM modes charge input/output characters. | Google documents Advanced API data-location options and service-specific data usage rules. | Good if glossary/domain terminology is more important than simplest setup. More cloud surface than this phase needs. |
| Azure AI Translator | Mature translation API with free/pay-as-you-go tiers and custom translator option. | S1 pay-as-you-go standard translation is character-metered; custom translation costs more. | Microsoft documents no-trace behavior for Translator submissions and optional CMK for supported tiers. | Viable alternative, especially if Microsoft/Azure is already a project standard. Not currently the simplest fit for this repo. |
| OpenAI API | Useful for style rewrite, QA checklist assistance, spoiler detection prompts or structured review notes. Less deterministic as a pure translation provider. | Token-metered; Batch API can reduce async draft cost. | OpenAI API data is not used for training by default, with abuse-monitoring/data controls documented. | Keep behind an optional `LocalizationDraftProvider` or `LocalizationReviewAssistant` port; do not make it required for v1 publication. |

Recommended provider decision for the phase:

- **Published v1 source:** QUEUE/2 curated/reviewed text stored in Postgres.
- **Draft provider default:** none required for execution; ship the provider port and CLI workflow so DeepL/OpenAI/Google can be attached later without changing runtime reads.
- **If provider-assisted drafts are included now:** implement DeepL as the first translation adapter because it is purpose-built for translation, character-metered and operationally smaller than a broad cloud translation stack.
- **OpenAI role:** optional reviewer/rewriter for natural PT-BR and checklist support, not the canonical translator and never a direct publisher.

## Data Model Recommendations

Add `catalog.game_localizations` with versioned rows:

- `id uuid primary key`
- `game_id uuid not null references catalog.games(id) on delete cascade`
- `locale varchar(16) not null`, initially `pt-BR`
- `version integer not null`
- `status varchar(24) not null` with at least `draft`, `review`, `published`, `rejected`
- `title text` optional if future localized titles are needed; description is the required Phase 02.1 field
- `description text`
- `source varchar(80) not null`, for example `queue2-curation`, `deepl-draft`, `openai-draft`
- `source_url text null`
- `rawg_description_hash text null` to know which RAWG source text a draft was based on without copying RAWG into localization rows by default
- `author_kind varchar(32)` and `author_id text null`
- `reviewer_kind varchar(32) null` and `reviewer_id text null`
- `review_notes text null`
- `quality_check jsonb not null default '{}'::jsonb`
- `created_at`, `updated_at`, `reviewed_at`, `published_at`

Constraints and indexes:

- Check status enum values.
- Check `locale = 'pt-BR'` or allow BCP-47 pattern; do not overgeneralize if only PT-BR ships.
- Unique `(game_id, locale, version)`.
- Partial unique index for a single current published row per `(game_id, locale)` if using `is_current = true`; otherwise read by `published_at desc, version desc`.
- Index `(game_id, locale, status, published_at desc)`.
- Runtime role gets `SELECT` only.
- Worker role gets insert/update for localization workflow.

Do not enable RLS on `catalog.game_localizations` unless catalog rows become private. The table is public catalog data, not duo-scoped. Least privilege and workflow separation matter more here than duo-membership policies.

## Sync And Operations Recommendations

The sync path should be a versioned worker/CLI command, not a browser action:

- Seed input lives in a tracked allowlist such as `apps/web/src/modules/catalog/infrastructure/seeds/coop-allowlist.ts` or `packages/db/src/seeds/catalog-coop-allowlist.ts`.
- Each seed entry includes RAWG id/slug, expected display name and QUEUE/2 curation fields: coop confirmation, player count, confirmation source, checked date and main-flow eligibility.
- RAWG updates only RAWG-owned facts: name, slug, cover, release, platforms, genres, source timestamps, RAWG URL and raw description storage.
- QUEUE/2-owned curation fields are preserved unless the allowlist explicitly carries an updated curated value.
- Dry-run mode prints structured item changes without writing.
- Apply mode writes in transactions and records run/item outcomes.

Recommended ops persistence:

- `ops.catalog_sync_runs`: `id`, `source`, `mode`, `started_at`, `finished_at`, `status`, `requested_by`, `dry_run`, `input_count`, `created_count`, `updated_count`, `failed_count`, `metadata jsonb`.
- `ops.catalog_sync_run_items`: `id`, `run_id`, `rawg_id`, `slug`, `status`, `game_id`, `changes jsonb`, `error_code`, `error_message`, `started_at`, `finished_at`.

These tables are catalog/ops operational data and should not require `duo_id`.

## Application And Module Shape

Keep all new behavior inside the catalog module public API:

- Extend `CatalogGameDetailRecord` with `localization: CatalogLocalizationRecord | null`.
- Add domain policies for localization state, such as `getLocalizationState` and `getCatalogSourceBreakdown`.
- Add repository reads that join the newest valid published `pt-BR` localization.
- Add worker/admin repository methods for `upsertLocalizationDraft`, `submitLocalizationForReview`, `publishLocalization` or a simpler seed import path if no admin UI is built.
- Replace `presentation/localized-descriptions.ts` with persisted seed migration and/or a seed command. The presentation layer should not own localized copy.

Routes should continue to import only from `apps/web/src/modules/catalog/index.ts`.

## UI Planning Notes

Cards should stay compact:

- Keep active RAWG attribution.
- Keep one short freshness line.
- Do not show the whole source table on every card.

Game detail should add a real `Fontes e frescor` block:

- `Dados e imagens: RAWG` with active RAWG link and relative freshness.
- `Descricao revisada: QUEUE/2` when a published localization exists.
- `Descricao em portugues ainda nao revisada` or equivalent when missing.
- `Tempo estimado: Sem fonte confiavel ainda` unless sourced.
- `Disponibilidade: Nao verificado` unless sourced.

Freshness must not depend only on color. The component can keep `data-freshness`, but every state needs text. Relative dates should be paired with accessible absolute dates through `dateTime`, visible title, or screen-reader text.

Do not show author/reviewer names in user-facing UI. Store them for audit.

## Security And Data Risks

- Provider secrets (`RAWG_API_KEY`, future `DEEPL_API_KEY`, `OPENAI_API_KEY`, Google/Azure credentials) must be server-only and covered by `scripts/check-secrets.mjs`.
- A translation provider adapter must live behind `server-only` and never be imported by Client Components.
- RAWG descriptions and provider drafts are untrusted text. Render as text, not HTML.
- Publication workflow must be explicitly admin/worker-only; normal authenticated users must not be able to mutate catalog localizations.
- Sync audit logs must redact request keys and avoid storing sensitive provider payloads beyond what is needed for review.
- Catalog sync should be idempotent. Retrying a run should not duplicate games, published localization rows or audit items.
- Integration tests should verify runtime role cannot insert/update localization or sync ops tables.

## Suggested Plan Slices

1. **Localization data contract and published reads.** Add schema/migration, Drizzle types, repository joins, domain fallback policy and seed migration path for current PT-BR copy.
2. **Auditable RAWG sync and provider decision.** Add curated allowlist sync command, dry-run/apply modes, ops run tables, curation-preserving upserts, provider decision documentation and optional draft-provider port.
3. **Source/freshness UI and verification.** Replace hardcoded presentation descriptions, add detail `Fontes e frescor`, keep cards compact, update tests for no English fallback, active RAWG attribution and secret boundaries.

Wave dependency should be strict: UI depends on persisted localization reads, and reliable sync depends on the curation-preserving repository contract.
