# Phase 03.3 Context: Performance De Producao E UX De Latencia

## Trigger

Production feedback on 2026-06-04 showed that QUEUE/2 feels slow in real use. The immediate symptom was `Adicionar a Wishlist` staying in a pending state for too long in `/app/catalogo`, but the underlying concern is broader: production latency, Server Actions, route refreshes, database query shape, and UI feedback need a dedicated phase before Phase 4 adds live sessions, scheduling and more write-heavy flows.

## Product Decision

Do not continue into Phase 4 until production performance is measured and improved. This phase is not decorative polish. It is a release-quality gate for the authenticated app experience.

## Initial Suspicions

- Catalog and Biblioteca Server Actions may wait on mutation, revalidation and full route refresh before the user sees completion.
- Authenticated routes may perform repeated server-side reads that are acceptable locally but slow in Vercel + Neon production.
- Catalog, Biblioteca and Descobrir list pages need query-count and `EXPLAIN` evidence, not only unit tests.
- Current E2E evidence is fixture-gated, so browser performance in production has not been honestly proven.
- Pending states exist, but they may communicate "stuck" instead of "saved locally, syncing".

## Phase Inputs

- Production URL: `https://queue-2.vercel.app`
- Recent symptom: Catalog Wishlist action remains visibly pending for too long.
- Relevant routes: `/app`, `/app/catalogo`, `/app/biblioteca`, `/app/descobrir`, `/app/jogo/[slug]`
- Relevant mutations: add to Wishlist, move Biblioteca status, Discovery decisions, Discovery handoff, Match Live and Quiz actions.
- Relevant constraints: secrets stay server-side; RLS/authorization cannot be weakened for speed; UI remains accessible and Portuguese-first.

## Planning Notes

The plan should start by measuring, not guessing. Capture production/preview traces and query counts first, then optimize the worst hot paths. UI latency should be improved with optimistic or redirect-free flows only where idempotency and server authority remain clear.
