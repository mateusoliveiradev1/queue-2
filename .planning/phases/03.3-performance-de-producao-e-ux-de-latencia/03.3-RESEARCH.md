# Phase 03.3 Research: Performance De Producao E UX De Latencia

## Research Question

What do we need to know to plan Phase 03.3 well?

This phase must turn the production complaint that `Adicionar a Wishlist` feels slow into a measured performance gate for the authenticated QUEUE/2 app. The plan should start with evidence, then optimize database/server hot paths and finally make latency visible to the user without weakening server authority, RLS or the modular architecture contract.

## Current Code Findings

### Route and mutation shape

- `/app/catalogo` reads `getDuoDashboard`, two catalog searches, then library statuses for visible cards. It renders Wishlist forms through `CatalogWishlistSubmitButton`.
- `/app/biblioteca` already uses the Phase 03.2 bounded `getLibraryQueue` contract, but repository hydration still calls per-row detail loaders through `hydrateLibraryRows`.
- `/app/descobrir` performs four server reads in parallel: deck, match history, live session and mood quiz status. Deck building calls catalog search first, then discovery read state for the candidate window.
- `/app/jogo/[slug]` loads catalog detail and then library detail for the current duo. Catalog detail hydration is per-game and includes source/freshness metadata.
- `addGameToWishlistAction`, `moveLibraryGameAction` and the Discovery actions currently end with redirects. `addGameToWishlistAction` also revalidates `/app`, `/app/biblioteca`, `/app/catalogo` and the return path before redirecting.
- Existing submit buttons expose `useFormStatus().pending`, but the user only sees success after the full server roundtrip, revalidation and redirect complete.

### Query hot-path risks

- `catalogRepository.searchGames` hydrates each returned game with separate platform, genre, localization, time estimate and availability queries. On `/app/catalogo`, 18 visible browse items plus suggestions can become a large query fan-out.
- `libraryRepository.getQueue` is bounded, but `hydrateLibraryRows` still hydrates each row one by one. The three slices (`nextQueue`, `playing`, page) can repeat per-game catalog facts and platform reads.
- `discoveryRepository.getReadState` batches decisions, library rows and matches for candidate IDs, which is better than per-card reads. The candidate source still depends on catalog search hydration.
- Existing DB tests verify RLS and some reviewed indexes for library/discovery, but Phase 03.3 needs query-count and `EXPLAIN` evidence tied to current route hot paths.

### Instrumentation gaps

- There is no app-level `useReportWebVitals` component or telemetry endpoint for TTFB/FCP/LCP/INP/CLS.
- There is no reusable safe duration wrapper for Server Actions, Route Handlers or domain use cases.
- Existing logs/audit helpers are security-focused. Performance events need redaction by construction and should avoid user emails, tokens, RAWG keys, session tokens, push endpoints and SQL parameter values.
- Playwright E2E suites already collect hydration errors and viewport evidence, but no route currently records TTFB, useful-content timing, first interaction latency, slow-network behavior or double-submit prevention as a performance gate.

## External Technical Notes

- Next.js `useReportWebVitals` reports TTFB, FCP, LCP, FID, CLS and INP and should live in a small client component imported by the root layout so the client boundary stays narrow.
- Vercel Speed Insights can track Core Web Vitals for production and preview deployments, but the project still needs local/CI browser gates because Speed Insights is observational and dashboard-based.
- Next.js Server Actions are public POST-reachable server functions, so every action must keep validation and authorization inside the server boundary. Optimistic UX cannot replace server authority.
- `revalidatePath` from a Server Function currently updates UI immediately for the viewed path and may refresh previously visited pages on later navigation. This makes broad revalidation a latency risk for user-perceived mutations.
- Next.js revalidation guidance prefers tag-based invalidation over broad path invalidation when possible because it is more precise and avoids over-invalidation.
- Neon connection pooling is appropriate for serverless apps with many short transactions. It is not a substitute for bounded query shapes and short transactions.
- Neon/Postgres query work should use `pg_stat_statements` or equivalent query capture to find candidates, then `EXPLAIN (ANALYZE, BUFFERS)` on representative data. Data-changing `EXPLAIN ANALYZE` must be wrapped in a transaction and rolled back.

## Planning Implications

### Wave 1 should measure before changing behavior

Create a thin observability layer:

- Client Web Vitals reporter for authenticated routes.
- Performance event route or server helper that accepts only allowlisted metric names and route labels.
- Server-side timer helpers for auth, database, external/cache, render/revalidation and total action duration.
- A `PERFORMANCE.md` or equivalent evidence artifact with baseline budgets for `/app`, `/app/catalogo`, `/app/biblioteca`, `/app/descobrir` and `/app/jogo/[slug]`.

The baseline must distinguish production/preview evidence from local evidence. Missing credentials or deployment URLs should be recorded as blockers, not pass evidence.

### Wave 2 should optimize database/query/server hot paths

Start from measured query counts and plans. Likely candidates:

- Batch catalog card hydration so platforms, genres, localization, time estimates and availability for a page are loaded in set-based queries keyed by game IDs.
- Batch library queue/detail hydration so catalog facts and platform keys are loaded for all slice/page rows together.
- Review Discovery deck candidate window and catalog hydration so it does not pay the full catalog detail cost for cards that will be filtered out.
- Add SQL shape tests and DB integration plan checks for catalog/library/discovery hot paths.
- Add migrations only when `EXPLAIN` proves missing indexes or bad plans. Use concurrent/index-safe migration patterns where applicable.

### Wave 3 should improve perceived mutation latency

User feedback must happen within about 100ms while preserving server authority:

- Replace redirect-only success paths with client-local optimistic/syncing states where the operation is reversible or idempotent.
- Keep native form progressive enhancement for no-JS and pre-hydration cases.
- Avoid claiming final success before the server responds. Use copy like "Salvo aqui, sincronizando..." and then authoritative success/error.
- Prevent double-submit through disabled states, idempotent server behavior and visible retry copy.
- Prefer precise revalidation or client refresh after the optimistic state, instead of broad `revalidatePath` calls before every redirect.

### Wave 4 should prove production-like behavior

Browser gates should cover:

- Desktop and mobile authenticated flows for Catalogo, Biblioteca, Descobrir and game detail.
- Slow network and at least one slow server/mutation simulation.
- Reduced motion equivalence.
- TTFB/resource timing capture and useful-content markers.
- First interaction latency for Wishlist, status move, Discovery decision, match handoff, Live and Quiz controls.
- Hydration errors, text overlap, focus-visible keyboard path and touch target checks.

## Risks and Constraints

- RLS and server authorization cannot be weakened for speed. Queries still run through `withAppUserTransaction` or explicit safe runtime context.
- Client optimism must not leak duo data, trust client-supplied `duoId`, or apply XP/status/future confirmation behavior locally.
- Performance logs must not contain sensitive values or raw SQL parameters.
- Current E2E fixtures are often missing. Plans must keep missing production/browser/database evidence as explicit blockers.
- Broad revalidation can create a false sense of correctness while making UX slower. Plans should force measurement before changing cache strategy.
- Performance-specific helpers should live under platform/runtime boundaries or module public contracts, not in route components as business logic.

## Recommended Plan Shape

1. Baseline and telemetry: Web Vitals, server timing helpers, performance evidence artifact and budgets.
2. Database/server hot paths: query counting, `EXPLAIN`, batched hydration, indexes/migrations only if evidence supports them.
3. Perceived latency UX: optimistic/syncing components and redirect-light mutation flows for primary actions.
4. Production-like gates: Playwright performance suite, reduced-motion/slow-network/double-submit checks and final evidence report.
