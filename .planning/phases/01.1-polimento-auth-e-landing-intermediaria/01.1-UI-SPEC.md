# Phase 01.1 UI Spec

## Design Intent

The interim public experience should feel like a deliberate QUEUE/2 product surface, not a placeholder and not the final launch landing. It should be compact, branded, accessible and useful while Phase 2 starts.

## Required UI Contracts

1. The home hero must make `QUEUE/2` the dominant first-viewport signal and show the literal product promise: a shared queue for exactly two players.
2. The home must expose three practical actions: Entrar, Criar conta and Parear dupla.
3. The home may preview the product ritual, but it must not fake catalog data or imply Phase 2+ features are already available.
4. Public auth surfaces must include a brand link to `/` with accessible name `Ir para a home QUEUE dois`.
5. The public intro panel should include a compact ritual/status composition so the left side does not read as empty filler.
6. The form panel should keep the `/2` mark as a compact badge, but the nearby composition must make the full brand visible through the linked wordmark.
7. All text must fit at mobile widths without using viewport-scaled font sizes outside existing clamp-based hero/display rules.
8. Focus rings and axe checks must remain clean.

## Visual Constraints

- Keep the sharp, editorial QUEUE/2 identity.
- Avoid a generic SaaS card stack and avoid arcade neon excess.
- No new decorative orb/bokeh backgrounds.
- Do not introduce new runtime dependencies.
- Reuse `@queue/ui` brand primitives and existing tokens.

## Acceptance

Passes when unit/brand tests cover the home and brand links, `pnpm --filter @queue/web typecheck` passes, and public Playwright accessibility can run without new violations when E2E env is configured.
