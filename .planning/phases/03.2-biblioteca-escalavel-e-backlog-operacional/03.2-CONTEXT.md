# Phase 03.2: Biblioteca Escalavel e Backlog Operacional - Context

**Gathered:** 2026-06-04T23:25:00-03:00
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase turns `/app/biblioteca` from a growing status-column page into the operational backlog surface for the duo. It must make the active queue scalable, searchable, paginated, URL-addressable, and usable on mobile while preserving the shared `/2` model.

The phase owns the Biblioteca information architecture, pagination/filtering contract, compact mobile layout, status action density, and the archive contract needed for later `Zerado`/`Dropado` behavior. It does not implement Phase 4 double confirmation, Principal/secondary ordering, sessions, progress, timer, chapters, milestones, scheduling, Hall/reviews, or roulette behavior.

</domain>

<decisions>
## Implementation Decisions

### Arquivo e ciclo de vida

- **D-01:** `Zerado` and `Dropado` leave the operational queue and belong in an Archive surface once Phase 4 unlocks those statuses.
- **D-02:** Archived games do not appear in the normal Discovery/recommendation cycle. They remain reachable only through explicit Archive, search, or game detail flows.
- **D-03:** Archive is reversible only by explicit user action. A duo can manually restore an archived game to an active queue status such as `Wishlist` or `Pausado`, but nothing returns automatically.
- **D-04:** Phase 03.2 must not show dead Archive controls before there are real archived items or real unlocked archive behavior. Prepare the structure/contract, but avoid disabled tabs, empty dead buttons, or placeholder UI that feels broken.

### Modelo da Biblioteca

- **D-05:** Biblioteca becomes an operational queue plus status views, not a three-column board that grows forever.
- **D-06:** The top of Biblioteca is `Proximos da fila`: a short actionable slice of what matters now, not a stats panel.
- **D-07:** `Jogando` gets a special area inside Biblioteca because it is active commitment, but it must not become the Phase 4 play dashboard.
- **D-08:** The `Jogando` area may show up to three active games, the active limit, status move/pause actions, and links to game detail. It must not show Principal/secondary roles, sessions, timer, chapters, subjective percentage, or milestones.
- **D-09:** Cards expose one primary action. Secondary status actions move into a menu, popover, or sheet so the UI stops showing every possible button at once.

### Crescimento e paginacao

- **D-10:** Biblioteca scales through pagination by view/status with filters encoded in the URL. Refresh, sharing, and browser back/forward should preserve the selected view.
- **D-11:** The 03.2 filters are essential only: status/view, text search by game name, common platform, and simple ordering.
- **D-12:** Simple ordering includes at least `mais recente`, `melhor match`, and `nome`. Do not duplicate the full Discovery filter set in Biblioteca.
- **D-13:** `Todas` means the active operational queue only: `Wishlist + Jogando + Pausado`. Archive is outside `Todas`.
- **D-14:** Page size should be light on mobile and denser on desktop: target 12 items per page on mobile and 24 on desktop when the layout supports it. The planner may choose the technical detection strategy, but the product intent is mobile-light, desktop-dense.

### Mobile e densidade visual

- **D-15:** Mobile cards are compact rows/cards with small cover, essential information, and one primary action. They should not use Discovery-sized hero cards.
- **D-16:** Mobile filters use a compact sticky bar with a filter button/sheet. Do not render all filters expanded above the content.
- **D-17:** Secondary actions open in a bottom sheet on mobile and a popover/menu on desktop.
- **D-18:** The compact card must show name, status, cover, common platforms, and a short match/compatibility cue. Do not shrink the current full card with all existing information.

### the agent's Discretion

- Choose exact query parameter names, default sort, page-size implementation, desktop density breakpoint, card composition, sheet/popover primitive, action labels, and empty-state copy.
- Choose whether `Proximos da fila` is computed by a dedicated application use case or derived from a paginated overview, as long as routes do not own business rules and repository reads do not hydrate an unbounded library.
- Choose the exact visual treatment for status chips, action menus, and pagination controls, preserving QUEUE/2 visual identity, accessibility, text fit, touch targets, and no dead controls.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `LibraryStatusControls` already centralizes status movement forms, but its current all-buttons layout is the source of visual clutter and should evolve into primary/secondary actions.
- `library-policy.ts` already owns valid statuses, active statuses, future terminal statuses, and the Jogando limit. Archive behavior should extend this policy rather than living in the route.
- `toLibraryOverviewView` already maps records into user-facing groups and counts. It should become pagination/count aware instead of deriving all counts only from fully hydrated arrays.
- Catalog and Discovery now have URL/page patterns that Biblioteca can reuse for stable navigation.
- `AppShell` and global mobile navigation already provide the mobile foundation from Phase 03.1; Biblioteca should work inside that shell rather than inventing a separate mobile nav.

### Established Patterns

- Authenticated app routes require verified sessions and redirect users without a ready duo.
- Routes compose module public APIs and server actions. Business rules belong in domain/application modules.
- Duo-scoped library data is protected by server authorization and RLS; pagination/filtering must preserve duo isolation.
- Source/freshness and match/compatibility explanations are short text facts, not fake numeric precision.
- Future states may appear in contracts, but UI must not expose disabled controls that look like broken features.

### Integration Points

- Add or adjust a library read use case for paginated views, counts, filters, sort, and `Proximos da fila`.
- Update `LibraryRepository` so the database can query only the requested slice instead of hydrating the entire duo library.
- Update `/app/biblioteca` search param parsing for URL-backed view, page, query, platform, and sort state.
- Replace column-board rendering with operational queue sections plus paginated result views.
- Replace always-visible status button clusters with primary action plus bottom sheet/popover secondary actions.
- Add browser/UI tests for mobile compact cards, sticky filter bar, no overlapping buttons, no disabled Archive controls, and stable URL pagination.

</code_context>

<specifics>
## Specific Ideas

- The current screenshot failure is the exact anti-pattern to avoid: three columns, empty panels, cards growing downward, many visible buttons, and no scalable path.
- `Proximos da fila` should answer "o que esta na nossa fila agora?", not "quantos cards existem?".
- `Todas` should feel like the active queue, not an archive dump.
- On mobile, the card should be compact enough for scanning many games but still recognizable by cover and compatibility cue.

</specifics>

- Replaying or automatically resurfacing archived games belongs to a future rule/phase, not 03.2.
- Phase 4 unlocks double confirmation for `Zerado` and `Dropado`, Principal/secondary roles, sessions, timer, progress, chapters, percentage, milestones, and scheduling.
- Phase 7 owns Hall/reviews/deep celebration of completed games.
