## Phase Interpretation

Phase 03.2 is not a new play/progress phase. It is the operational backlog phase that makes `/app/biblioteca` scalable and useful before Phase 4 unlocks the richer `Jogando Agora` system.

The page must stop behaving like a growing three-column board. It needs a bounded, URL-addressable queue surface with `Proximos da fila`, a special but limited `Jogando` area, paginated status/search views, compact mobile cards, and dense status actions that do not expose dead future controls.

## Current Implementation Findings

### Data and application layer

- `apps/web/src/modules/library/application/get-library-overview.ts` exposes only `getLibraryOverviewUseCase(userId, repository)`.
- `apps/web/src/modules/library/application/ports.ts` defines `LibraryOverviewRecord` as full grouped arrays for `wishlist`, `jogando`, and `pausado`.
- `apps/web/src/modules/library/infrastructure/library-repository.ts` implements `getOverview` by:
  - loading membership,
  - loading all member platforms,
  - loading all duo library games through `getLibraryGames`,
  - hydrating each game individually through `hydrateLibraryGame`,
  - grouping all active statuses in memory.
- The current repository read is unbounded for a growing backlog and produces N+1 hydration behavior. Phase 03.2 should replace or supplement it with bounded SQL reads that return only the requested page, plus counts and `Proximos da fila`.
- Existing indexes already include `app_duo_library_games_duo_status_idx (duo_id, status, updated_at DESC)`, which supports status-filtered recent ordering. Search by game name and sort by name/match may need query review, but no schema migration appears required for the first scalable contract unless query plans prove otherwise.

### Domain and archive contract

- `apps/web/src/modules/library/domain/library-policy.ts` already defines:
  - active Phase 2 statuses: `wishlist`, `jogando`, `pausado`,
  - future confirmation statuses: `zerado`, `dropado`,
  - `JOGANDO_LIMIT = 3`,
  - future-status blocking for Phase 2 moves.
- Phase 03.2 should make the archive policy explicit in domain/application terms: active queue equals `wishlist + jogando + pausado`; archive equals `zerado + dropado` once Phase 4 can create those states.
- The UI must not display dead archive tabs, disabled archive buttons, or fake empty archive surfaces before real archive rows and behavior exist.

### Current route and UI

- `apps/web/src/app/app/biblioteca/page.tsx` currently renders:
  - platform picker and common-platform summary,
  - status tabs,
  - all selected status sections in a `.library-board`,
  - a locked future status section for `Zerado` and `Dropado`.
- `LibraryGameCard` lives inside the route, uses a 96px cover row, and renders `LibraryStatusControls` plus `Abrir detalhe`.
- `LibraryStatusControls` currently renders all status buttons and two disabled buttons: `Zerado bloqueado` and `Dropado bloqueado`. This directly conflicts with Phase 03.2 D-04 and should be replaced with one primary action plus secondary actions in a compact menu/sheet pattern.
- `apps/web/src/app/globals.css` has existing library styles but keeps the board/card/control structure. Discovery already introduced useful sheet/orbit patterns (`discovery-filter-sheet`, `discovery-search-sheet`, `app-bottom-nav`) that Biblioteca can reuse stylistically without copying Discovery-specific layout.

### URL and pagination precedent

- `apps/web/src/app/app/catalogo/page.tsx` already has a simple URL-backed page contract with `q`, `pagina`, `limit + 1`, `offset`, and `buildCatalogPath`.
- `apps/web/src/modules/catalog/application/search-catalog.ts` and `catalog-repository.ts` provide a simple limit/offset pattern and bounded `clampLimit`.
- Biblioteca should use its own route parameter parser instead of putting parsing rules directly across JSX. Parameters can be chosen by the executor, but the plan should require stable URL state for view/status, search query, common-platform filter, sort, and page.

## Impeccable Use

The user confirmed Phase 03.2 should use Impeccable for UI/UX. The GitHub reference describes Impeccable as a frontend design vocabulary/rubric with domain references for typography, color/contrast, motion, spatial design, interaction, responsive design, and UX writing, plus deterministic anti-pattern rules and commands such as `shape`, `critique`, `audit`, `polish`, `clarify`, and `adapt`.

For this phase, use Impeccable as an acceptance rubric, not as a package dependency:

- `shape`: Biblioteca must become an operational backlog surface, not a board of growing columns.
- `clarify`: copy must distinguish active queue, `Jogando`, archive contract, and future Phase 4 behavior without dead controls.
- `adapt`: mobile must use compact rows/cards, sticky filter bar, bottom-sheet-like secondary actions, and no Discovery-sized cards.
- `audit`: tests and browser review must catch overlap, invisible focus, unbounded lists, broken URL state, disabled archive controls, and text overflow.
- `polish`: UI must remain QUEUE/2 brutalist/editorial and calm-utility, not generic SaaS or arcade neon.

Do not install Impeccable or add it to project dependencies unless the user explicitly asks later.

## Recommended Planning Breakdown

### Plan 01 - Scalable library read contract

Create a bounded library read contract before rebuilding the route. This should add typed query inputs, active/archive policy helpers, page-size/sort validation, counts, `Proximos da fila`, and repository queries that avoid hydrating the entire duo library.

Why first: the route cannot honestly become scalable while it depends on `getOverview` full hydration.

### Plan 02 - Operational Biblioteca route

Recompose `/app/biblioteca` around `Proximos da fila`, a limited `Jogando` area, and a paginated active queue view. Search params should preserve view, page, query, platform, and ordering through refresh/back/share. Remove dead locked-status UI from the route.

Why second: once bounded data exists, the page can stop being a board.

### Plan 03 - Compact cards, filters and status actions

Extract or refactor Biblioteca presentation components so cards are compact, mobile-ready, and action-dense. Replace all-buttons status controls with one primary action plus secondary actions rendered as a menu/sheet-compatible surface. Keep valid mutations server-authoritative through existing server actions.

Why third: action density and mobile card ergonomics depend on the new IA and route states.

### Plan 04 - Browser, accessibility and regression evidence

Add focused UI, domain/application, E2E, accessibility, architecture and source-guard coverage. Browser checks should verify desktop/mobile/reduced-motion, URL state, pagination, no dead archive controls, no overlapping status controls, no unbounded repository calls, and no Phase 4-only features.

Why fourth: Phase 03.2 is a UX gate before Jogando, so skipped browser evidence must be recorded honestly rather than treated as pass.

## Risks and Controls

| Risk | Control |
|------|---------|
| Unbounded library reads remain hidden behind view-model helpers. | Add tests/source guards proving route uses the new bounded use case and repository has a paginated query path. |
| UI accidentally exposes Phase 4 controls. | Source/UI tests must reject `Zerado bloqueado`, `Dropado bloqueado`, dead Archive tabs, sessions, timer, chapters, percentage, milestones, Principal and secondary copy. |
| Route owns business rules while parsing URL state. | Keep route parsing in a small route params helper and delegate queue/read rules to library application/domain. |
| Sort by match becomes expensive or ambiguous. | Use existing qualitative match score/read model for ordering in application/repository where feasible; if SQL cannot sort by full computed score cheaply, plan an explicit deterministic fallback without fake precision. |
| Mobile filter/action surfaces become too dense. | Use Impeccable `adapt`/`harden` criteria: sticky compact filter bar, minimum 44px targets, text wrapping, no nested cards, no all-filters-expanded first fold. |
| Archive contract is overbuilt before Phase 4. | Prepare domain/read contract and explicit route behavior, but do not render empty/dead archive UI until real archived items/behavior exist. |
